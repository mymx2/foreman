# 进程调度延迟分析 — 使用指南

## 简介

`sched-latency` 技能用于分析进程的调度延迟（scheduling latency），帮助运维人员快速定位调度层面的性能瓶颈。

调度延迟包括两部分：
- **唤醒延迟（wake-up delay）**：进程被唤醒到实际获得 CPU 运行的等待时间
- **抢占延迟（preemption delay）**：进程处于可运行状态但被其他进程占用 CPU 而排队等待的时间

无需记忆复杂的命令参数——只需告诉 AI 你想分析哪个进程，它会自动选择最合适的工具、执行采集、解读结果，并给出优化建议。

## 适用场景

| 场景 | 典型表现 | 使用本 Skill 的价值 |
|------|---------|-------------------|
| 进程响应变慢 | 业务延迟升高，但进程 CPU 使用率并不高 | 定位是否因等待 CPU 调度导致响应变慢 |
| 延迟毛刺排查 | P99/max 延迟偶发飙高 | 通过阈值过滤和调用栈采集，定位延迟毛刺的根因 |
| CPU 密集场景 | 多进程/多容器共享 CPU，互相争抢 | 统计唤醒延迟和抢占延迟的分布，评估 CPU 竞争程度 |
| 容器 CPU 限流 | cgroup CPU quota 配置后进程变慢 | 分析 RD（调度延迟）状态占比，确认是否为 CPU 限流导致 |
| NUMA 调度分析 | 跨 NUMA 节点调度导致的额外开销 | 通过 task-state 和中间事件分析跨节点调度情况 |
| 偶现问题捕获 | 调度延迟问题难以复现 | 部署长期监控脚本，持续运行数天自动捕获异常 |

## 使用方法

### 第 1 步：触发 Skill

在 WorkBuddy 输入框中用自然语言描述需求即可，例如：

- "帮我分析 nginx 进程的调度延迟"
- "分析 PID 12345 的调度延迟"
- "看看 java 进程是不是调度慢导致卡顿"
- "进程响应慢，排查一下调度延迟"
- "帮我部署调度延迟监控脚本，监控 3 天"

### 第 2 步：AI 自动执行分析

AI 会按以下逻辑自动运行：

1. **确认目标进程和阈值** — 通过 `ps`/`pgrep` 找到进程 PID，从用户描述中提取或推荐延迟阈值
2. **选择最佳工具** — 按优先级 perf-prof rundelay > perf sched 自动选择
3. **渐进式分析** — 先基础统计，再按线程统计、阈值过滤、详细跟踪、调用栈采集，逐步深入
4. **输出报告** — 以表格形式展示延迟分布，附分析结论和优化建议
5. **建议长期监控** — 调度延迟问题通常为偶现，建议部署后台监控脚本持续捕获

> 💡 如果服务器上没有 perf-prof，AI 会询问你是否需要编译安装（推荐）。perf-prof 的 rundelay 分析器支持唤醒延迟和抢占延迟的分开统计、延迟分布、按线程维度统计等高级功能。

### 第 3 步：查看分析报告

报告包含以下内容：

- **调度延迟 Top 线程表格**：按延迟排序，分别展示唤醒延迟和抢占延迟的 p50/p99/max
- **延迟分布分析**：唤醒延迟 vs 抢占延迟的分布特征
- **RD 状态占比**：进程在调度延迟状态的时间占比（如果执行了 task-state 分析）
- **严重程度判定**：基于阈值参考表的延迟等级评估
- **分析结论和优化建议**：针对具体问题的改进方向

## 分析工具说明

本 Skill 支持以下工具，按优先级自动选择：

| 工具 | 优先级 | 优点 | 局限 |
|------|--------|------|------|
| **perf-prof rundelay** | ⭐ 推荐 | 开销低、实时处理、支持唤醒/抢占延迟分开统计、延迟分布、阈值过滤、详细跟踪、调用栈 | 需手动安装 |
| **perf-prof task-state** | 辅助 | 分析进程状态分布（R/RD/S/D），直观看 RD 占比 | 辅助工具，需配合 rundelay 使用 |
| **perf sched** | 备选 | 内核自带、无需额外安装 | 只有平均/最大延迟，无法区分唤醒/抢占延迟，无分位数统计 |

> ⚠️ 调度延迟分析**没有兜底方案**：strace 无法分析调度延迟。perf-prof 和 perf 都不可用时会直接报错终止。

## 长期监控模式

调度延迟问题往往是偶现的，短时间采集几乎不可能抓到现场。本技能支持生成后台监控脚本，持续运行数天来捕获延迟毛刺。

### 工作原理

监控脚本启动两个 perf-prof 通道并行运行：
1. **统计摘要通道**：持续输出每秒的延迟统计（按线程维度），记录全局趋势
2. **详细事件通道**：仅记录超阈值的延迟事件，同时采集调用栈和同 CPU 中间事件，用于根因分析

### 使用方式

- "帮我生成 nginx 进程的调度延迟监控脚本，监控 3 天"
- "部署一个后台脚本监控 PID 12345 的调度延迟，阈值 10ms"

### 日志文件

监控脚本会在日志目录（默认 `/var/log/sched-latency-monitor`）下生成：

| 文件 | 内容 |
|------|------|
| `summary.log` | 每秒的延迟统计摘要（按线程维度） |
| `detail.log` | 超阈值事件的详细信息 + 调用栈 + 同 CPU 中间事件 |
| `monitor.log` | 脚本自身运行日志 |
| `monitor.pid` | 监控脚本主进程 PID |

### 分析监控结果

监控日志收集完成后，可以将日志内容提供给 AI 进行分析：

```
提示词：帮我分析 /var/log/sched-latency-monitor/ 下的调度延迟监控日志
```

AI 会从 summary.log 和 detail.log 中提取延迟高发时段、分析调用栈定位根因，并输出完整的分析报告。

## 实践案例

### 案例 1：微服务 P99 延迟飙高 — 定位 cgroup CPU 限流

**场景背景**：容器化部署的 Java 微服务，P99 延迟从 10ms 飙到 100ms+，但容器 CPU 使用率只有 60%。

**问题描述**：业务高峰期延迟毛刺频繁出现，容器的 CPU limit 设置为 2 核。

**解决方案**：

```
提示词：帮我分析 java 进程的调度延迟，任务经常超时 10ms
```

AI 使用 perf-prof rundelay 采集后发现：

| 延迟类型 | p50 | p99 | max |
|---------|-----|-----|-----|
| 唤醒延迟 | 0.5ms | 15.2ms | 85.3ms |
| 抢占延迟 | 0.2ms | 8.7ms | 42.1ms |

结合 task-state 分析，RD 状态占比 18%，确认是 cgroup CPU quota 限流导致。建议将 CPU limit 从 2 核调整为 4 核，或优化应用减少 CPU 消耗。调整后 P99 延迟降至 8ms。

### 案例 2：偶现延迟毛刺 — 部署长期监控捕获根因

**场景背景**：Nginx 进程偶尔出现 50ms+ 的延迟毛刺，发生频率约每天 2-3 次，短时间采集抓不到。

**问题描述**：实时分析 30 秒未观察到超阈值延迟。

**解决方案**：

```
提示词：帮我部署 nginx 进程的调度延迟监控脚本，监控 3 天，阈值 50ms
```

AI 部署监控脚本后，3 天共捕获 7 次超 50ms 延迟事件。从 detail.log 的调用栈分析发现，延迟期间同 CPU 上有一个周期性的内核线程长时间占用 CPU。建议调整该内核线程的 CPU 亲和性或降低其优先级。

## 常见误区与 Tips

### ❌ 误区 1：perf-prof 不可用时不尝试安装就直接降级

perf-prof rundelay 的分析能力远优于 perf sched（支持延迟分布、唤醒/抢占分开统计、阈值过滤等）。AI 会先尝试编译安装 perf-prof，只有确实无法安装才降级到 perf。

### ❌ 误区 2：只看短时间采集结果就下结论

调度延迟问题通常为偶现，10-90 秒的采集结果不具备统计充分性。如果短期采集未复现问题，**务必考虑部署长期监控**。

### ❌ 误区 3：只关注平均延迟，忽略长尾

平均延迟 1ms 看似正常，但如果 max 延迟达到 100ms，说明存在严重的长尾延迟。重点关注 **p99 和 max** 指标。

### 💡 Tip 1：提供具体延迟时间可跳过阈值确认

如果你在描述中包含具体的延迟数值（如"任务经常超时 10ms"），AI 会直接使用该值作为分析阈值，无需额外确认。

### 💡 Tip 2：结合 task-state 综合分析

perf-prof task-state 可以查看进程在 R（运行）、RD（调度延迟）、S（睡眠）、D（不可中断睡眠）各状态的时间占比。RD 占比高说明调度延迟严重，S 占比高说明进程在等待 IO 或锁。

### 💡 Tip 3：高频场景记得增大 ringbuffer

如果目标进程调度事件频率很高（如高并发 Web 服务器），可能出现事件丢失。perf-prof 的 stderr 会输出 `lost xx events on` 警告，此时需增大 `-m` 参数（如 `-m 256`）。

## 关键阈值参考

| 指标 | 正常 | 需关注 | 严重 |
|------|------|--------|------|
| p50 | <1ms | 1-5ms | >5ms |
| p99 | <10ms | 10-50ms | >50ms |
| max | <50ms | 50-100ms | >100ms |

## 注意事项

- perf-prof 和 perf 需要 **root 权限**
- 调度延迟分析**不提供兜底方案**（strace 无法分析调度延迟），perf-prof 和 perf 都不可用时会直接报错终止
- 高频调度场景下需注意 perf ringbuffer 大小，避免事件丢失（`-m 256`）
- 内核 4.14+ 的 `prev_state` 字段类型有变化，perf-prof 会自动适配
- 确保 sched tracepoint 可用（debugfs 已挂载）
- 长期监控脚本依赖 perf-prof，不支持 perf 降级
