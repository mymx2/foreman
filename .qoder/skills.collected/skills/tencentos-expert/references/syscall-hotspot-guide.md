# 系统调用热点分析 — 使用指南

## 简介

`syscall-hotspot` 技能帮助你快速找出进程中耗时最多、调用最频繁或出错最多的系统调用，从而定位性能瓶颈。

无需记忆复杂的命令参数——只需告诉 AI 你想分析哪个进程，它会自动选择最合适的工具、执行采集、解读结果，并给出优化建议。

## 适用场景

| 场景 | 典型表现 | 使用本 Skill 的价值 |
|------|---------|-------------------|
| 进程响应变慢 | 接口延迟升高，但 CPU/内存并不高 | 定位是否卡在某个系统调用上（如 futex、epoll_wait） |
| IO 密集型应用 | 大量 read/write，磁盘 IO 高 | 统计 IO 系统调用的频率和延迟分布 |
| 系统调用报错 | 应用日志有 errno 错误 | 找出哪些系统调用在报错及错误次数 |
| 性能对比 | 优化前后效果确认 | 对比系统调用的调用次数和耗时变化 |
| 进程阻塞排查 | 进程 hang 住不动 | 发现是否卡在某个阻塞型系统调用 |

## 使用方法

### 第 1 步：触发 Skill

在 WorkBuddy 输入框中用自然语言描述需求即可，例如：

- "帮我分析 nginx 进程的系统调用热点"
- "分析 PID 12345 的系统调用耗时"
- "看看 java 进程哪个系统调用最慢"
- "排查进程系统调用报错原因"
- "分析 mysql 的 IO 相关系统调用性能"

### 第 2 步：AI 自动执行分析

AI 会按以下逻辑自动运行：

1. **确认目标进程** — 通过 `ps`/`pgrep` 找到进程 PID
2. **选择最佳工具** — 按优先级 perf-prof > perf > strace 自动选择
3. **采集数据** — 运行分析命令，采集系统调用的次数、耗时、错误等指标
4. **输出报告** — 以表格形式展示热点 Top 10，附分析结论和优化建议

> 💡 如果服务器上没有 perf-prof，AI 会询问你是否需要编译安装（推荐）。安装后分析效果最好、开销最低。

### 第 3 步：查看分析报告

报告包含以下内容：

- **热点 Top 10 表格**：列出调用次数、总耗时、平均耗时、最大耗时、错误次数
- **分析结论**：高频热点、延迟毛刺、错误等问题的总结
- **优化建议**：针对具体问题的改进方向

## 分析工具说明

本 Skill 支持三种工具，按优先级自动选择：

| 工具 | 优先级 | 优点 | 局限 |
|------|--------|------|------|
| **perf-prof** | ⭐ 推荐 | 开销低、实时处理、支持延迟分布和阈值过滤 | 需手动安装 |
| **perf** | 备选 | 内核自带、功能全面 | 统计维度不如 perf-prof 丰富 |
| **strace** | 兜底 | 使用简单、几乎所有系统都有 | 开销大，不适合生产环境长时间使用 |

## 实践案例

### 案例 1：Nginx 响应变慢 — 定位 epoll_wait 延迟毛刺

**场景背景**：Web 运维人员发现 Nginx 服务偶尔出现响应延迟飙高，但 CPU 和内存使用率正常。

**问题描述**：P99 延迟从平时的 5ms 飙到 50ms+，无明显规律。

**解决方案**：

```
提示词：帮我分析 nginx worker 进程的系统调用热点，重点看有没有延迟毛刺
```

AI 使用 perf-prof 采集后发现：

| 排名 | 系统调用 | 调用次数 | 平均耗时 | 最大耗时 | 错误次数 |
|------|---------|---------|---------|---------|---------|
| 1 | epoll_wait | 8523 | 1.2ms | 48.7ms | 0 |
| 2 | read | 12045 | 5.3us | 2.1ms | 0 |
| 3 | write | 11892 | 8.7us | 3.4ms | 23 |

**结论**：`epoll_wait` 最大耗时 48.7ms 远大于平均 1.2ms，存在延迟毛刺。进一步排查发现是内核 timer 精度问题。

### 案例 2：Java 应用 IO 性能分析

**场景背景**：Java 微服务在高并发时吞吐量下降。

**问题描述**：TPS 从 5000 降到 2000，怀疑 IO 瓶颈。

**解决方案**：

```
提示词：分析 java 进程的 IO 相关系统调用性能
```

AI 采集数据后定位到 `write` 调用频率极高（每秒 50000+ 次），每次写入数据量很小，建议合并写入或增大缓冲区。优化后 TPS 恢复到 4800。

## 常见误区与 Tips

### ❌ 误区 1：直接用 strace 分析生产进程

strace 基于 ptrace 机制，会显著影响目标进程性能。**生产环境请优先使用 perf-prof**（开销低 1-2 个数量级）。

### ❌ 误区 2：忽视事件丢失警告

如果 perf-prof 输出 `lost xx events on`，说明 ringbuffer 不够大，统计数据不完整。需要加大 `-m` 参数（如 `-m 256`）。

### ❌ 误区 3：只看调用次数，不看耗时

调用次数最多的系统调用不一定是瓶颈。真正的瓶颈往往是**总耗时最大**或**最大耗时远超平均**的系统调用。

### 💡 Tip 1：先总览再聚焦

建议先做一次不带过滤的全量分析，找到热点后再用阈值过滤（`--than`）或事件过滤器深入分析。

### 💡 Tip 2：对比分析效果好

优化前后各做一次分析，用数据对比证明优化效果。

### 💡 Tip 3：注意架构差异

不同 CPU 架构（x86_64、ARM64）的系统调用编号不同。perf-prof 和 strace 会自动显示名称，但如需手动过滤编号需确认架构。

## 注意事项

- perf-prof 和 perf 需要 **root 权限**
- strace 使用 ptrace 机制，对进程性能影响较大，**不适合生产环境长时间跟踪**
- 不同 CPU 架构（x86_64、ARM64、RISC-V）的系统调用编号不同
- 高频系统调用场景下需注意 perf ringbuffer 大小，避免事件丢失
- 分析前确认目标进程会持续运行足够长时间
